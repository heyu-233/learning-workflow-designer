# FreeRTOS 轻量示例参考答案

## 第 1 章 题 1

正确顺序是：硬件初始化 -> 创建任务 -> 启动调度器 -> 任务循环运行并通过阻塞 API 让出 CPU。

创建任务必须发生在调度器启动之前，因为调度器启动后会开始选择就绪任务运行；如果没有任务被创建，系统没有可调度的应用任务。任务循环中如果没有阻塞或延时，高优先级任务可能持续占用 CPU，导致低优先级任务长期无法运行。

## 第 2 章 题 1

方案 A 会持续消耗 CPU，方案 B 让任务进入阻塞态，等事件到达后再进入就绪态。方案 B 更符合 FreeRTOS 的事件驱动调度方式。方案 A 只有在等待时间极短、系统没有其他重要任务、或非常早期的临时验证中才可能暂时可用。

## 第 2 章 题 2

任务函数通常不能直接 `return`，因为任务由调度器管理，返回会破坏任务生命周期。任务如果需要主动结束，应调用 `vTaskDelete(NULL)` 删除自身。

## 第 3 章 题 1

两个同优先级就绪任务会在时间片机制下轮流获得 CPU。某个任务调用 `vTaskDelay()` 后进入阻塞态，直到延时到期再回到就绪态等待调度。

## 第 4 章 题 1

错误位置是中断服务函数中直接调用普通任务上下文 API `xQueueSend()`。更合适的链路是：中断服务函数 -> `xQueueSendFromISR()` -> 必要时请求上下文切换 -> 任务调用 `xQueueReceive()` -> 处理消息。错误 API 可能导致调度状态异常、临界区行为不符合预期，甚至产生难以复现的运行问题。

## 第 5 章 题 1

二值信号量适合表达单一事件到达或资源释放；事件组适合表达多个事件位的组合关系。等待多个事件同时成立时，优先考虑事件组。

## 第 6 章 题 1

这是优先级反转问题。FreeRTOS 互斥锁可通过优先级继承，让持锁的低优先级任务临时继承等待者的高优先级，从而尽快释放资源。

## 第 7 章 题 1

`heap_1` 只支持分配，不支持释放，因此不适合频繁创建和删除不同大小对象。可以考虑支持释放与合并的 heap 方案，例如 `heap_4`。

## 第 8 章 题 1

可检查栈高水位、开启栈溢出检测钩子、增大任务栈观察现象是否消失、查看 HardFault 堆栈和任务切换点、减少局部大数组或递归调用。

## 第 9 章 题 1

最小方案可以记录每个任务的心跳时间戳、队列当前长度或发送失败次数、关键同步点等待超时日志，并在调试接口周期性输出任务状态。

## 第 10 章 题 1

先确认串口中断确实触发，再确认 ISR 使用 `xQueueSendFromISR()` 成功投递消息，然后检查控制任务是否阻塞在队列接收、是否被更高优先级任务饿死，再检查互斥锁或信号量是否长期占用，最后用日志定位卡在哪个边界。
